# Smart Context - 智能上下文管理

通过双轨记忆、增量合并和三步清洗法解决长对话中的注意力稀释问题。

## 核心理念

**问题**：增量日志积累 → 90% 噪音 → 注意力稀释 → 忘记目标 / 重复踩坑

**解法**：增量合并 + 结构化快照，而非全量重新压缩

```
增量日志（几万 Token）──Incremental Merge──▶ 热轨快照（目标 ~6k Token）
```

**优化目标**：**Tokens per Task**（完成整个任务的总 Token 消耗），不是 Tokens per Request。
过度压缩导致信息丢失 → 后续重读文件浪费更多 Token → 得不偿失。

## Automatic Activation Triggers

This skill should be automatically activated when:

**Keywords:**
- compact, 压缩, 上下文, context
- 快照, snapshot, checkpoint
- 墓碑, tombstone, 死胡同, 避坑
- 注意力稀释, context window
- 热轨, 冷轨, hot track, cold track
- 之前试过, 历史方案, 为什么不行, 失败原因

**Example User Messages:**
- "上下文太长了" / "执行压缩" / "/compact"
- "生成一个快照" / "/context-snapshot"
- "之前试过什么方案？" / "我们尝试过什么？"
- "那个方案为什么不行？" / "XX方案失败的原因是？"
- "查一下冷轨" / "/query-cold"

**When NOT to Activate:**
- 用户只是问问题，不涉及上下文管理或历史查询
- 用户在讨论其他不相关的话题

## 双轨制记忆

### 热轨 (Hot Track)
- **目标容量**：~6,000 Token（软目标，按恢复成本动态调整）
- **形式**：`.agent/HOT_TRACK.md`
- **内容**：原始意图 + 项目状态 + 接口骨架 + 文件操作日志 + 避坑墓碑 + 关键决策 + 下一步
- **特点**：增量合并生成，手动加载

### 冷轨 (Cold Track)
- **容量**：无限
- **形式**：`.agent/COLD_TRACK.md`
- **内容**：完整历史 + 墓碑详情
- **特点**：按需查询（通过对话触发）

## 工作流程

### 完全手动流程

```
1. /context-snapshot  →  增量合并生成快照（三步清洗法 + 自验证）
2. /compact           →  执行压缩
3. /load-context      →  新会话需要时手动加载快照
```

> **注意**：采用完全手动流程，避免旧快照污染新会话。

## 四步清洗法

### Step 1: 无损提取 — 抽骨架

**目标**：从杂乱对话中提取"沉淀下来的资产"

**提取规则**：

| 内容类型 | 提取方式 | 保留 |
|----------|----------|------|
| 函数定义 | 只保留签名 | `def calc(a: int, b: int) -> int` |
| 类定义 | 只保留属性和方法签名 | 删除方法体 |
| API 端点 | 只保留路由和参数定义 | 删除处理逻辑 |
| 数据模型 | 只保留字段定义 | 删除默认值逻辑 |

**执行过程**：
1. 扫描所有代码相关内容（工具调用、代码块）
2. 识别最终版本的接口（中间版本丢弃）
3. 提取函数签名、类型定义、API 路由
4. 生成骨架，不包含实现细节

### Step 2: 有损修剪 — 立墓碑

**目标**：识别失败尝试，生成避坑指南

**死胡同识别模式**：

```markdown
信号词：
- 假设提出："试试"、"也许"、"尝试一下"
- 尝试实施："我改了"、"修改为"、"实现了一下"
- 明确失败："不行"、"还是报错"、"没解决"
- 放弃方案："换一种"、"改用"、"算了"

匹配模式：[假设] + [尝试] + [失败] + [放弃] = 死胡同
```

**墓碑格式**：

```markdown
| 死胡同 | 失败原因 | 日期 |
|--------|----------|------|
| [方案名称] | [一句话说明为什么不行] | [日期] |
```

**执行过程**：
1. 遍历对话历史，识别死胡同模式
2. 为每个死胡同生成墓碑（< 50 字）
3. 将详细失败过程写入冷轨
4. 保留墓碑到热轨

### Step 3: 增量合并 — 组装快照

**目标**：基于已有快照进行增量合并，而非全量重新生成

**增量合并流程**（Anchored Iterative Summarization）：

```
已有 HOT_TRACK.md（anchor）
       +
新对话内容（自上次快照后）
       │
       ▼
  ┌─────────────────────────┐
  │ 1. 读取已有快照作为 anchor │
  │ 2. 只分析新增对话内容      │
  │ 3. 提取新信息（骨架/墓碑等）│
  │ 4. 合并到对应 section      │
  │ 5. 更新进度状态            │
  │ 6. 保留原始意图不变        │
  └─────────────────────────┘
       │
       ▼
  更新后的 HOT_TRACK.md
```

**首次快照**：如果不存在已有快照，则从完整对话历史生成初始快照。

**合并规则**：

| Section | 合并策略 |
|---------|----------|
| 原始意图 | **永不修改**（首次写入后冻结） |
| 项目状态 | 替换为最新状态 |
| 文件操作日志 | **只追加，不删除** |
| 接口骨架 | 追加新增接口，更新已有接口的最新版本 |
| 避坑墓碑 | 追加新墓碑 |
| 关键决策 | 追加新决策 |
| 下一步 | 替换为最新计划 |

**Token 估算规则**：

```python
def estimate_tokens(text: str) -> int:
    """
    混合 Token 估算：
    - 英文/代码/符号：1 Token ≈ 4 字符
    - 中文/日文/韩文：1 Token ≈ 1.5 字符
    """
    chinese_chars = count_chinese(text)  # Unicode: \u4e00-\u9fff
    other_chars = len(text) - chinese_chars
    tokens = (chinese_chars / 1.5) + (other_chars / 4)
    return int(tokens)
```

**裁剪策略**（接近目标 Token 时）：

采用 **恢复成本加权** 裁剪，而非简单优先级：

| 信息类型 | 恢复成本 | 裁剪优先级 |
|----------|----------|-----------|
| 原始意图 + 阻塞 | 极高（无法自动恢复） | 最后裁剪 |
| 文件操作日志 | 高（需重读所有文件） | 压缩路径但保留列表 |
| 避坑墓碑 | 高（可能重蹈覆辙） | 压缩原因，保留死胡同名 |
| 接口骨架 | 中（可从代码恢复） | 删除参数详情 |
| 关键决策 | 中（可从 git 恢复） | 可删除 |
| 进度概览 | 低（可重新评估） | 最先裁剪 |

### Step 4: Probe 自验证

**目标**：验证快照质量，确保关键信息未在压缩中丢失

对生成的快照执行四类 Probe 检查：

| Probe 类型 | 验证问题 | 期望 |
|------------|----------|------|
| **Recall** | 原始用户意图是什么？快照中是否保留？ | 精确匹配 |
| **Artifact** | 本次会话读/写/创建了哪些文件？快照中是否列出？ | 完整列表 |
| **Continuation** | 下一步该做什么？快照中是否明确？ | 可执行步骤 |
| **Decision** | 关键技术决策及理由？快照中是否记录？ | 决策+原因 |

**执行过程**：
1. 生成快照后，针对每个 Probe 提问
2. 从快照内容中寻找答案
3. 如果某个 Probe 验证失败：
   - 回到对话历史补充缺失信息
   - 重新合并到对应 section
   - 再次验证直到通过
4. 输出验证结果统计

**输出格式**：

```markdown
## 自验证结果

| Probe | 状态 | 说明 |
|-------|------|------|
| Recall | ✅ | 原始意图：实现 JWT 认证 |
| Artifact | ⚠️ 补充 | 已回补 2 个遗漏文件 |
| Continuation | ✅ | 下一步：修复剩余 2 个测试 |
| Decision | ✅ | 3 个决策已记录 |
```

**输出格式**：

## 最佳实践

### 何时压缩

- 上下文超过 70% 窗口时
- 完成一个阶段性任务后
- 开始新的功能开发前
- 遇到"注意力稀释"症状时

### 如何写高质量墓碑

Bad: 用 Session 不行
Good: Session 存储登录态 → 分布式部署时同步问题，改用 JWT

### 如何维护热轨

- 每次压缩前使用 /context-snapshot 增量合并
- 手动编辑添加重要信息
- 定期清理过时的决策
- 原始意图 section 一旦写入不要修改

### 增量合并的注意事项

- 文件操作日志只追加不删除，保持完整追踪链
- 原始意图首次写入后冻结，后续合并不修改
- 接口骨架保留最新版本，旧版本自动替换
- 多次快照后注意 Token 膨胀，必要时按恢复成本裁剪

## 技术限制

1. **PreCompact Agent Hook 不支持** - 在非 REPL 环境下无法使用 agent 类型 hook
2. **无自动注入** - 为避免旧快照污染新会话，采用完全手动流程
3. **Token 估算不精确** - 使用字符估算，非精确计算
4. **冷轨查询基于 Grep** - 不支持语义搜索
5. **Probe 验证非 LLM 评分** - 使用规则匹配验证，非 LLM-as-judge 模式
