# Context Snapshot - 手动生成快照

## 使用方式

```
/context-snapshot
```

## 功能

手动触发四步清洗法，通过增量合并生成热轨快照和冷轨归档，并执行 Probe 自验证，但不执行压缩。

**适用场景**：
- 预览快照内容
- 在压缩前生成快照
- 调试四步清洗法效果

---

## 执行步骤

### Step 0: 检查已有快照

1. 检查 `.agent/HOT_TRACK.md` 是否存在
   - **存在**：读取作为 anchor，进入增量合并模式
   - **不存在**：进入首次生成模式
2. 确保 `.agent/` 目录存在（不存在则自动创建）

### Step 1: 无损提取 — 抽骨架

**目标**：从杂乱对话中提取"沉淀下来的资产"

**提取规则**：

| 内容类型 | 提取方式 | 示例 |
|----------|----------|------|
| 函数定义 | 只保留签名 | `def calc(a: int, b: int) -> int` |
| 类定义 | 只保留属性和方法签名 | 删除方法体 |
| API 端点 | 只保留路由和参数定义 | 删除处理逻辑 |
| 数据模型 | 只保留字段定义 | 删除默认值逻辑 |
| 文件操作 | 记录路径 + 操作类型 | 读/写/创建/删除 |

**执行过程**：
1. 扫描所有代码相关内容（工具调用、代码块）
2. 识别最终版本的接口（中间版本丢弃）
3. 提取函数签名、类型定义、API 路由
4. **提取所有文件操作记录**（读取、修改、创建的文件路径）
5. 生成骨架，不包含实现细节

### Step 2: 有损修剪 — 立墓碑

**目标**：识别失败尝试，生成避坑指南

**死胡同识别模式**：

```
信号词：
- 假设提出："试试"、"也许"、"尝试一下"
- 尝试实施："我改了"、"修改为"、"实现了一下"
- 明确失败："不行"、"还是报错"、"没解决"
- 放弃方案："换一种"、"改用"、"算了"

匹配模式：[假设] + [尝试] + [失败] + [放弃] = 死胡同
```

**墓碑格式**：

```markdown
| 死胡同 | 失败原因 | 日期 |
|--------|----------|------|
| [方案名称] | [一句话说明为什么不行] | [日期] |
```

**执行过程**：
1. 遍历对话历史，识别死胡同模式
2. 为每个死胡同生成墓碑（< 50 字）
3. 将详细失败过程写入冷轨
4. 保留墓碑到热轨

### Step 3: 增量合并 — 组装快照

**目标**：基于已有快照进行增量合并，而非全量重新生成

**首次生成模式**：
- 从完整对话历史生成初始快照
- 原始意图 section 首次写入

**增量合并模式**：
- 读取已有 HOT_TRACK.md 作为 anchor
- 只分析自上次快照后的新对话内容
- 按合并规则更新各 section

**合并规则**：

| Section | 合并策略 |
|---------|----------|
| 原始意图 | **永不修改**（首次写入后冻结） |
| 项目状态 | 替换为最新状态 |
| 文件操作日志 | **只追加，不删除** |
| 接口骨架 | 追加新增接口，更新已有接口的最新版本 |
| 避坑墓碑 | 追加新墓碑 |
| 关键决策 | 追加新决策 |
| 下一步 | 替换为最新计划 |

**Token 估算规则**：

```
混合 Token 估算：
- 英文/代码/符号：1 Token ≈ 4 字符
- 中文/日文/韩文：1 Token ≈ 1.5 字符

估算公式：
  chinese_chars = count_chinese(text)
  other_chars = len(text) - chinese_chars
  tokens = (chinese_chars / 1.5) + (other_chars / 4)
```

**裁剪策略**（接近目标 Token 时）：

采用 **恢复成本加权** 裁剪：

| 信息类型 | 恢复成本 | 裁剪优先级 |
|----------|----------|-----------|
| 原始意图 + 阻塞 | 极高（无法自动恢复） | 最后裁剪 |
| 文件操作日志 | 高（需重读所有文件） | 压缩路径但保留列表 |
| 避坑墓碑 | 高（可能重蹈覆辙） | 压缩原因，保留死胡同名 |
| 接口骨架 | 中（可从代码恢复） | 删除参数详情 |
| 关键决策 | 中（可从 git 恢复） | 可删除 |
| 进度概览 | 低（可重新评估） | 最先裁剪 |

**执行过程**：
1. 组装热轨快照（原始意图 + 项目状态 + 接口骨架 + 文件操作日志 + 墓碑 + 决策 + 下一步）
2. 估算 Token 数
3. 如果超限，按恢复成本加权裁剪
4. 写入 `.agent/HOT_TRACK.md`
5. 追加完整历史到 `.agent/COLD_TRACK.md`

### Step 4: Probe 自验证

**目标**：验证快照质量，确保关键信息未在压缩中丢失

| Probe 类型 | 验证问题 | 期望 |
|------------|----------|------|
| **Recall** | 原始用户意图是什么？快照中是否保留？ | 精确匹配 |
| **Artifact** | 本次会话读/写/创建了哪些文件？快照中是否列出？ | 完整列表 |
| **Continuation** | 下一步该做什么？快照中是否明确？ | 可执行步骤 |
| **Decision** | 关键技术决策及理由？快照中是否记录？ | 决策+原因 |

**执行过程**：
1. 对生成的快照执行四类 Probe 检查
2. 从快照内容中寻找答案
3. 如果验证失败 → 回到对话历史补充缺失信息 → 重新合并
4. 输出验证结果统计

```markdown

_生成时间: YYYY-MM-DD HH:MM_
_快照版本: v{N}（增量合并）_
_上次快照: YYYY-MM-DD HH:MM（首次生成则标记为"初始"）_

---

## 0. 原始意图

> [用户的原始需求，首次写入后永不修改]

---

## 1. 项目状态

### 当前进度
| 模块 | 状态 | 说明 |
|------|------|------|
| ... | 进行中/已完成/阻塞 | ... |

### 当下阻塞
- **问题**: [描述]
- **下一步**: [计划]

---

## 2. 已确定接口（骨架）

### {文件名}
[接口签名，无实现体]

---

## 3. 文件操作日志

| 文件路径 | 操作 | 关键变更 | 日期 |
|----------|------|----------|------|
| ... | 读/写/创建/删除 | ... | ... |

---

## 4. 避坑墓碑

| 死胡同 | 失败原因 | 日期 |
|--------|----------|------|
| [方案] | [一句话] | [日期] |

---

## 5. 关键决策

| 决策 | 理由 | 日期 |
|------|------|------|
| [选择] | [原因] | [日期] |

---

## 6. 下一步

1. [具体可执行的步骤 1]
2. [具体可执行的步骤 2]

---
*Token 数: ~{N}*
*验证: Recall ✅ Artifact ✅ Continuation ✅ Decision ✅*
```

## 与 /compact 的区别

| 操作 | /context-snapshot | /compact |
|------|-------------------|----------|
| 生成快照 | ✓ | ✗ |
| 增量合并 | ✓ | ✗ |
| Probe 验证 | ✓ | ✗ |
| 执行压缩 | ✗ | ✓ |
| 适用场景 | 生成/更新快照 | 压缩上下文 |

**推荐流程**：
1. `/context-snapshot` - 增量合并生成快照（含自验证）
2. `/compact` - 执行压缩
3. `/load-context` - 新会话需要时加载快照

---

## 注意事项

- 首次生成会从完整对话历史创建快照，后续使用增量合并
- 原始意图 section 一旦写入不再修改
- 文件操作日志只追加不删除，保持完整追踪链
- Token 目标约 6,000（软限制，按恢复成本动态调整）
- 冷轨无大小限制，完整记录历史
